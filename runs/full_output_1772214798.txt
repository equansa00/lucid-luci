LUCI Code should be designed with a focus on efficiency, safety, and flexibility. Here’s a detailed specification based on the core capabilities and upgrades you’ve outlined:

### Core Capabilities

#### Project Context
- **Tree Structure**: LUCI should display a tree view of the project directory, allowing users to navigate through files and folders.
- **Ignore Rules**: Users can define ignore rules (e.g., `.git`, `node_modules`) to exclude certain directories or files from operations.
- **Selective File Loading**: Operations like search, edit, and run commands should be limited to selected files rather than the entire codebase.

#### Tools
- **Read**: Display the content of a file in a readable format.
- **Search**: Perform text searches within the project, optionally filtering by file type or directory.
- **Edit (Patch-Based)**: Allow users to make changes using patch-based edits. LUCI should show diffs and require approval for each change.
- **Create/Delete**: Provide options to create new files, directories, and delete existing ones.
- **Run Commands**: Execute shell commands within the project context, with options to run locally or through an API.
- **Git Status/Diff/Commit**: Integrate with Git to show status, view diffs between commits, and commit changes.

#### Agent Loop
- **Plan → Act → Observe → Iterate**: Implement a loop where LUCI plans actions based on the current state of the project, executes those actions, observes the outcome, and iterates accordingly.
- **Budgets and Stop Conditions**: Allow users to set budgets for operations (e.g., time, resources) and define stop conditions based on specific criteria.

#### Safe Edits
- **Show Diffs**: Display a diff view before applying any changes.
- **Approval Required**: Require manual approval for each edit to ensure safety and prevent unintended modifications.

#### Sessions
- **Per-Project State**: Maintain state-specific data for each project, including open files, cursor positions, and recent actions.
- **Long-Term Memory Integration**: Optionally integrate with LUCI’s long-term memory for persistent project data.

#### Streaming Terminal UI
- **Tokens Stream**: Display terminal output in real-time as tokens are generated.
- **Tool Events**: Show events related to tools (e.g., file creation, deletion) within the terminal UI.
- **Diff Viewer**: Integrate a diff viewer within the terminal to show changes side-by-side.

#### Offline-First
- **Ollama/Local Models**: Use Ollama/local models for operations when offline. Optionally integrate with an API later for online capabilities.

### Better Than Upgrades

#### Deterministic Patch Engine
- **Unified Diffs**: Ensure that patch-based edits are deterministic and consistent.
- **AST-Aware Edits**: Where possible, use Abstract Syntax Trees (ASTs) to make more intelligent edits that respect the structure of the code.

#### Policy Controls
- **Allowlist Commands**: Restrict users to a predefined list of allowed commands for security and compliance.
- **Path Sandboxing**: Limit file operations to specific directories based on user-defined policies.
- **Network Off by Default**: Ensure network access is disabled by default, with explicit options to enable it.

#### PR Pipeline
- **Branch Naming**: Enforce consistent naming conventions for branches (e.g., `feature/`, `bugfix/`).
- **Commit Messages**: Require specific formats or templates for commit messages.
- **Changelog**: Automatically generate changelogs based on the commits made in a branch.
- **CI Run Orchestration**: Integrate with CI/CD tools to automate testing and deployment processes.

### Implementation

To achieve these capabilities, LUCI should be built using a combination of frontend technologies (React for the UI) and backend services (Python/Docker for project management and operations). The agent loop can be implemented as a state machine that continuously runs in the background, observing changes and making adjustments based on predefined rules.

For offline functionality, LUCI should include a local database or caching mechanism to store project data and allow operations to proceed even when not connected to the internet. The deterministic patch engine can be built using libraries like `diff-match-patch` for unified diffs and tools like `astor` for AST-aware edits.

By following this specification, LUCI will provide a powerful and flexible development environment that is both efficient and safe, making it an upgrade over existing solutions.
